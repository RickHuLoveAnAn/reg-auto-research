## 做得好的

1. **Temperature=0 确定性评分**
   - 切换前: τ 在 0.2-0.6 之间剧烈振荡
   - 切换后: τ 稳定在 0.8-1.0
   - 结论: LLM 评分固有的随机性会破坏 ranking 稳定性

2. **5维 → 4维 精简**
   - 结构清晰度在 7 次运行中被 Agent 多次归零（无区分贡献）
   - 删除后系统更稳定，权重分布更合理

3. **实操落地性维度设计**
   - pairwise_acc=0.88，是唯一接近完美的维度
   - 直接贡献了 τ=1.0 的达成

4. **Rubric 泛化（behavioral anchors）**
   - 初始 rubric 过于抽象（如"有一定战略视角"）
   - 收紧为具体行为描述后（如"具体数字预测"），战略洞察力 pairwise_acc 从 0.13 提升到 0.24

5. **Git 版本控制**
   - 每次 τ 提升自动 commit
   - 可追溯、可回滚，迭代过程完整记录

## 有待改善

### 1. Agent 过度修正（最严重）

**问题**: Agent 15 次把某个维度权重直接归零，其中战略洞察力被归零 9 次。

**根因**: Agent 的 deviation 报告中看到某个维度 pairwise_acc=0.0，就判定它"无效"并把权重设为 0。但实际上：
- pairwise_acc=0.0 可能只是 LLM 评分的随机波动
- 该维度对整体排序仍有间接贡献
- Agent 不理解"组合效应"——多维度加权后的排序效果不等于各维度效果之和

**后果**: 每次权重归零后，τ 大概率下降，git checkout 丢弃改动，但浪费了迭代机会。

### 2. 3维阶段浪费 57 次迭代

Run 1-5 共 57 次迭代，τ=1.0 为 0 次。

**根因**:
- temperature=0.1 导致评分不稳定（主要）
- 战略趋势 rubric 区分度差（次要）

**教训**: 后续应直接用最优配置（4维 + temperature=0）起步。

### 3. 评分波动

即使 temperature=0，pairwise_acc 仍有波动（实操落地性 0.6 ↔ 0.9）。

**根因**: rubric 的边界描述仍有模糊地带，LLM 在边界分数上可能有不同解读。

### 4. Golden set 规模小

仅 5 篇文章，10 对组合。任何一对判断错误就会损失 10% 的 pairwise_acc。

## 后续提升建议

| 优先级 | 问题 | 解决方案 |
|--------|------|---------|
| **P0** | Agent 把有效维度归零 | 修改 agent prompt：禁止将任何维度权重设为 0；要求权重调整幅度 <30% |
| **P0** | 起步阶段浪费迭代 | 后续直接用 4维 + temperature=0 起步，跳过 3维探索 |
| **P1** | 实操落地性 0.6↔0.9 波动 | 收紧 rubric 边界，例如明确"指出 3 条以上合规红线"得 10 分 |
| **P2** | Golden set 只有 5 篇 | 扩充到 10-20 篇，提升排序置信度 |
| **P2** | Agent 只调权重不改 rubric | 让 Agent 也优化 rubric 内容，而不仅是权重 |
| **P3** | 无迭代预算控制 | 增加 max_iterations 参数，避免无限迭代 |

### Agent Prompt 改进建议

```
注意：
- 禁止将任何维度的权重设为 0.0 或 1.0
- 每次权重调整幅度不得超过 30%
- 如果某个维度 pairwise_acc 偏低（<0.5），优先优化其 rubric，而非降低权重
- 组合效果不等于各维度效果之和：不要因为单个维度偶发低分就放弃它
```

```

  论证严谨性  25%  (pairwise_acc=0.80)
  内容纯度    25%  (pairwise_acc=0.70)
  实操落地性  30%  (pairwise_acc=0.90)
```
